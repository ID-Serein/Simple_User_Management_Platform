# Flask 用户管理平台安全修复成果

## 1. 修复概述

本次修复针对前期代码审计报告中发现的认证、会话、密码、CSRF、Debug 暴露和安全响应头等问题进行了加固。修复后，项目不再包含硬编码管理员密码，不再保存或回显明文密码，旧的 `dev-key-2025` session 伪造路径已失效，登录和退出流程也加入了 CSRF 与限速保护。

当前项目仍适合作为轻量 Flask 示例使用，但运行时需要通过环境变量或 `.env` 文件提供密钥和初始化管理员密码。

## 2. 修复项对照

| 原漏洞 | 修复结果 | 关键文件 |
| --- | --- | --- |
| Flask `secret_key` 硬编码 | 改为从 `FLASK_SECRET_KEY` 读取，长度不足会拒绝启动；本地开发无配置时使用随机临时密钥 | `app.py` |
| 默认管理员账号密码泄露 | 删除登录页 HTML 注释和源码中的默认明文密码 | `templates/login.html`、`app.py` |
| 明文密码存储和回显 | 初始化密码只生成哈希保存，登录使用哈希校验，页面不返回密码或哈希 | `app.py`、`templates/index.html` |
| Debug 模式对外监听 | 默认关闭 Debug，默认监听 `127.0.0.1`；Debug 模式禁止绑定非本地地址 | `app.py` |
| 登录暴力破解风险 | 加入 IP + 用户名维度的失败次数限制，超过阈值返回 `429` | `app.py` |
| 缺少 CSRF 防护 | 登录和退出表单加入 CSRF Token，退出改为 POST | `app.py`、`templates/*.html` |
| 缺少安全响应头 | 增加 CSP、`X-Frame-Options`、`X-Content-Type-Options`、`Referrer-Policy` 等响应头 | `app.py` |
| 依赖和配置不可复现 | 新增依赖文件和环境变量示例 | `requirements.txt`、`.env.example` |
| 用户数据可能误提交 | 将本地用户数据目录加入 Git 忽略 | `.gitignore` |

## 3. 密码处理成果

修复前，用户密码以明文形式硬编码在源码中，并且登录后会被页面回显。

修复后：

- 源码中不再保存真实用户明文密码；
- 初始化管理员密码通过 `INITIAL_ADMIN_PASSWORD` 提供；
- 密码最短要求为 12 位；
- 服务端使用 `generate_password_hash()` 生成密码哈希；
- 登录时使用 `check_password_hash()` 校验；
- 页面和响应中不再输出密码或 `password_hash`；
- 本地用户数据默认保存到 `data/users.json`，且 `data/` 已加入 `.gitignore`。

说明：密码哈希主要防止数据泄露后直接暴露明文密码，不能单独防止在线爆破。因此本次同时加入了登录失败限速。

## 4. 新增配置说明Skill 编写与审查规则

本文件用于约束本仓库内 Skill 的编写、修改、检查和 PR review 行为。任何 AI 工具、自动化 Agent 或人工协作者在处理本仓库 Skill 时，都应按以下触发条件执行对应流程。

## 触发条件

1. 当新增、修改、迁移或整理任意 Skill 时，触发“编写前执行流程”和“Skill 完成后检查流程”。
2. 当新增或修改 `SKILL.md`、`tools/`、readme、脚本、模板、字典引用或路径变量时，触发 `skills编写规则.md` 的对应规则检查。
3. 当准备提交 PR 时，触发“PR 前检查流程”。
4. 当用户要求 review PR，或需要在 Gitee/GitHub 等平台发表评论时，触发“Review 输出格式”和“行级 Review 要求”。
5. 当发现规范不满足时，优先修复后再继续验证；无法安全修复时，在 PR 说明或 review 评论中明确阻塞原因、影响范围和建议处理方式。

## 通用要求

1. 编写、修改、迁移或整理任意 Skill 前，必须先读取仓库根目录的 `skills编写规则.md`，并在编写过程中严格按该文件执行。
2. 每次写完或改完任意 Skill 后，必须立即按 `skills编写规则.md` 逐行检查。
3. 每次提 PR 前，必须再按 `skills编写规则.md` 对本次变更范围做一次总检查。
4. 必须把 `skills编写规则.md` 中的每条强制规则、检查清单和禁止项转化为当前变更的检查项，逐项核对，不得只做概括性浏览。
5. 每次完成检查后，必须逐条输出 `skills编写规则.md` 中相关规则是否满足；不能幻想、不能跳项、不能用“整体符合”替代逐条结论。
6. 检查范围至少包含本次新增、修改、移动、删除涉及到的 Skill 目录；如果规范要求检查同目录所有文件，则必须覆盖同目录的 `SKILL.md`、`tools/`、readme、脚本和模板文件。

## 编写前执行流程

1. 读取规范：
   - 必须读取 `skills编写规则.md`。
   - 未读取该文件前，不得开始新增、修改、迁移或整理 Skill。

2. 按规范编写：
   - 编写 `SKILL.md`、`tools/`、readme、脚本、模板、字典引用和路径变量时，必须同步对照 `skills编写规则.md`。
   - 不得写完后再凭印象补救；编写过程中就要遵守规范。

## Skill 完成后与 PR 前检查流程

1. 确认变更范围：
   - 使用 `git status --short`、`git diff --name-only` 或等价命令识别本次 PR 涉及的文件。
   - 按目录判断 Skill 类型：产品型 nuclei Skill、通用技术类 Skill、父级分类 Skill 或其他类型。

2. 逐行执行规范检查：
   - 读取 `skills编写规则.md` 全文。
   - 按文件中的顺序逐条核对规则。
   - 对每个受影响 Skill 检查 `SKILL.md` 的 frontmatter、`description`、正文长度、工具清单、路径写法、父子目录关系和禁止内容。
   - 对 `tools/` 下的 `.yaml` / `.yml`、`.py`、`*_readme.md`、`Dockerfile` 等文件按规范逐项检查。
   - 输出逐条检查结果，至少包含：规则原文或规则摘要、检查对象、是否满足、证据或修复动作。

3. 验证工具可用性：
   - 如果存在新增或修改的 `.py` 脚本，必须实际执行可用性验证。
   - 优先按对应 `_readme.md` 中的命令执行；如果 readme 命令不符合规范，先修复 readme，再按修复后的命令验证。
   - 普通 Python 工具优先使用规范要求的 `uv run --directory` 形式；pocsuite 工具使用 `pocsuite -r` 形式。
   - 如脚本需要目标地址、凭据或外部服务，至少执行 `--help`、参数解析、导入检查或无害 dry-run；并在 PR 说明中写明未执行真实探测的原因。
   - 验证失败必须修复；不能修复时不得声称脚本可用。

4. 验证 nuclei 模板：
   - 如果受影响目录包含新增或修改的 `.yaml` / `.yml` 模板，必须执行 `nuclei -validate -t <tools目录>/`。
   - 若本地没有 nuclei，必须记录环境缺失，并至少执行 YAML 解析检查；同时明确说明 YAML 解析不能替代 nuclei 校验。

5. 检查重复和路径问题：
   - 检查同一产品、同一品牌、同一分类目录下是否存在重复工具。
   - 检查 readme 和 `SKILL.md` 中重复出现的完整路径是否按规范抽成有语义前缀的变量。
   - 删除、合并或重命名工具后，必须同步更新 `SKILL.md` 和 readme。

6. 最终 PR 说明：
   - 列出实际检查过的 Skill 目录。
   - 列出执行过的验证命令及结果。
   - 列出已修复的问题。
   - 列出仍存在的风险或未能执行的验证，并说明原因。

## Review 输出格式

每次审查 Skill、输出问题、或在 PR 中发表评论时，必须使用稳定格式。不要只写概括性意见，不要只说“不符合规范”。每个问题都必须能让作者直接定位、理解并修改。

当用户要求 review PR，或需要在 Gitee/GitHub 等平台上发表评论时，必须优先使用对应文件、对应 diff 行的行级评论。能挂到具体新增/修改行的问题，不要发总评评论。

### 单个问题格式

```markdown
问题 <序号>：<简短标题>

- 位置：`<文件路径>:<行号>`（必要时写多个位置；没有行号时说明原因）
- 违反规则：引用或概括 `skills编写规则.md` 中对应规则
- 当前问题：说明当前代码/文档为什么不满足规则，必须基于实际文件内容或命令结果
- 修改建议：给出可执行修改方式，尽量说明应该改成什么
- 验证方式：说明修改后需要执行的检查命令或人工复核点
```

### 行级 Review 要求

- 能对应到 diff 中某一行的问题，必须评论到该文件对应行，例如 `databases/redis/tools/pyproject.toml:2`。
- 每条行级评论只说一个具体问题，避免把多个无关问题塞进同一条评论。
- 行级评论正文必须直接说明“这行违反了什么规则”和“建议如何改”，格式如下：

```markdown
这行不符合 `<规则摘要>`：<当前问题>。

建议：<具体修改建议>。
验证：<修改后需要执行的检查或复核点>。
```

- 如果同类问题分散在多行，可以选择最有代表性的一行挂评论，并在正文中说明“同类位置还包括 ...”。
- 如果问题无法挂到具体 diff 行，例如缺少文件、目录结构归并问题、整目录校验失败，优先挂到最相关的文件行；仍然无法定位时，只有在用户明确允许总评的情况下，才在 PR 总评中说明原因，并给出可搜索的文件路径或目录。

### PR 总评格式（仅在用户明确要求时使用）

默认不要发布 PR 总评、综述、审查结论、主要问题汇总、规则逐项检查汇总。Review 平台上的评论应尽量全部是单条问题、单个文件、单个行号。

只有在用户明确要求“输出总结 / 发总评 / 给整体结论”，或平台/问题确实完全无法挂到任何文件行时，才允许使用 PR 总评。跨文件问题也应优先挂到最相关的新增/修改行，并在正文中说明“同类位置还包括 ...”，而不是发一大段综述。

```markdown
审查结论：<通过 / 需要修改后再合并 / 阻塞合并>

检查范围：
- `<Skill 目录或文件>`

发现的问题：

问题 1：<简短标题>

- 位置：`<文件路径>:<行号>`
- 违反规则：<规则摘要>
- 当前问题：<基于实际内容说明>
- 修改建议：<可执行建议>
- 验证方式：<命令或复核点>

规则逐项检查：
- `SKILL.md description`：<满足 / 不满足 / 不适用>，证据：<简述>
- `SKILL.md 长度`：<满足 / 不满足 / 不适用>，证据：<字符数或文件>
- `Nuclei YAML 校验`：<满足 / 不满足 / 不适用>，证据：<命令结果>
- `Python 脚本 readme`：<满足 / 不满足 / 不适用>，证据：<文件或缺失项>
- `Python 依赖固化`：<满足 / 不满足 / 不适用>，证据：<pyproject/Dockerfile 情况>
- `路径变量`：<满足 / 不满足 / 不适用>，证据：<变量或路径>
- `字典位置与引用`：<满足 / 不满足 / 不适用>，证据：<路径>
- `目录结构归并`：<满足 / 不满足 / 不适用>，证据：<目录情况>

已执行验证：
- `<命令>`：<通过 / 失败 / 未执行，原因>

未执行验证：
- `<验证项>`：<原因>
```

### 输出要求

- 问题必须按严重程度排序：阻塞合并的问题在前，普通建议在后。
- 每个问题必须包含 `位置`、`违反规则`、`当前问题`、`修改建议`、`验证方式` 五项。
- 涉及多处同类问题时，可以合并为一个问题，但 `位置` 必须列出代表性行号，并说明同类范围。
- 行号必须来自实际读取的文件内容或 diff，不得幻想。
- 如果无法获得行号，必须写明“未能获得行号”的原因，并用文件路径和可搜索文本定位。
- 规则逐项检查不能省略；不适用时写“不适用”并说明原因。
- 在 Gitee/GitHub PR 评论中也必须使用同一格式。
- 在 Gitee/GitHub PR 评论中，默认只发对应行的行级评论；不要发布“审查结论 / 主要问题 / 已验证 / 规则逐项”这类综述评论。
- 跨文件问题优先挂到最相关的文件行；只有完全无法定位到任何 diff 行，且用户允许总评时，才发总评。
- 如果 review 内容过长，拆成多条行级评论；不要先发总评。

## 工作原则

- 不要跳过规范文件；没有读规范，不得提 PR。
- 不要在未读取 `skills编写规则.md` 的情况下开始编写 Skill。
- 不要等到 PR 前才第一次检查；每写完或改完一个 Skill 就要按规范检查一次。
- 不要幻想检查结果；必须基于实际读取、实际命令输出、实际文件内容逐条判断。
- 不要只检查改动文件；规范要求同目录检查时，必须覆盖同目录所有相关文件。
- 不要把通用技术类 Skill 当成产品型 nuclei Skill 批量重写。
- 不要在没有执行验证的情况下写“已验证可用”。
- 不要引入无关重构、格式化或大范围改写。
- 对用户已有改动保持尊重，只修复与本次 PR 和规范检查直接相关的问题。

新增 `.env.example`，运行前可复制为 `.env` 并按需修改：

```text
FLASK_SECRET_KEY=replace-with-a-random-secret-of-at-least-32-characters
INITIAL_ADMIN_USERNAME=admin
INITIAL_ADMIN_PASSWORD=replace-with-a-strong-password-at-least-12-characters
USER_STORE_PATH=data/users.json
FLASK_DEBUG=0
FLASK_RUN_HOST=127.0.0.1
```

注意事项：

- 不要提交 `.env`；
- `FLASK_SECRET_KEY` 应使用 32 位以上高强度随机字符串；
- 生产环境应开启 HTTPS，并设置 `SESSION_COOKIE_SECURE=1`；
- 生产环境应设置 `SECURITY_ENABLE_HSTS=1`。
